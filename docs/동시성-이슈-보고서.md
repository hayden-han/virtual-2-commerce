## 1. 문제상황

### 1.1 개요

시스템 내 동시성 처리가 필요한 영역에서 **4건의 동시성 이슈**가 존재

| 영역                   | 문제 유형       | 현재 상태     |
|----------------------|-------------|-----------|
| 잔액 충전                | Lost Update | **이슈 있음** |
| 재고 감소 (1개 상품)        | Overselling | ✅ 해결됨     |
| 재고 감소 (대량 주문)        | Overselling | ✅ 해결됨     |
| 쿠폰 발급 (OnePerMember) | 중복 발급       | **이슈 있음** |

### 1.2 현재 적용된 동시성 제어

| 영역         | 보호 메커니즘                                 | 비고         |
|------------|-----------------------------------------|------------|
| 잔액 차감 (주문) | Member Lock (SELECT FOR UPDATE)         | 동일 사용자만 보호 |
| 쿠폰 발급 (수량) | CouponIssuance Lock (SELECT FOR UPDATE) | 수량 초과 방지   |
| 쿠폰 사용 (주문) | Member Lock (SELECT FOR UPDATE)         | 간접 보호      |

---

### 1.3 상세 문제 분석

#### 1.3.1 잔액 충전 동시성 이슈

**문제 코드 위치**: `MemberBalanceService.recharge()`

**시나리오**
```
초기 상태: 사용자 A의 잔액 10,000원

[동시 요청]
- 요청 1: 5,000원 충전
- 요청 2: 3,000원 충전

기대 결과: 최종 잔액 18,000원 (10,000 + 5,000 + 3,000)
실제 결과: 13,000원 또는 15,000원 (한 충전 금액 유실)
```

**발생 원인**
```
시간 →
Thread 1: READ(10,000) ───────────────────── WRITE(15,000)
Thread 2: ────── READ(10,000) ── WRITE(13,000)

결과: Thread 1의 충전(5,000원)이 Thread 2에 의해 덮어씀 (Lost Update)
```

**비즈니스 영향**
- 고객이 충전한 금액이 유실됨
- 금전적 손실 발생
- 고객 신뢰도 하락

---

#### 1.3.2 재고 감소 동시성 이슈

**문제 코드 위치**: `ProductSummaryService.reduceStock()`

**시나리오 (1개 상품)**
```
초기 상태: 상품 재고 1개

[동시 요청 - 다른 사용자]
- 사용자 A: 1개 주문
- 사용자 B: 1개 주문

기대 결과: 1명 성공, 1명 실패 (재고 부족)
실제 결과: 2명 모두 성공 (Overselling)
```

**발생 원인**
```
시간 →
Thread 1 (User A): LOCK(member_A) ── 재고확인(1) ── 차감 ── COMMIT
Thread 2 (User B): LOCK(member_B) ── 재고확인(1) ── 차감 ── COMMIT

문제: Member Lock은 사용자별로 분리되어 있어,
      다른 사용자 간의 동시 재고 접근을 보호하지 못함
```

**비즈니스 영향**
- 재고보다 많은 주문 접수
- 배송 불가 상황 발생
- 주문 취소 처리 필요
- 고객 불만 및 신뢰도 하락

---

#### 1.3.3 쿠폰 발급 동시성 이슈 (OnePerMember)

**문제 코드 위치**: `CouponIssuanceFacade.issue()` 내 정책 검증

**시나리오**
```
초기 상태: OnePerMember 정책 쿠폰, 사용자 C에게 미발급

[동시 요청 - 동일 사용자]
- 요청 1: 쿠폰 발급 (idempotency-key-1)
- 요청 2: 쿠폰 발급 (idempotency-key-2)

기대 결과: 1건 성공, 1건 실패 (이미 발급됨)
실제 결과: 2건 모두 성공 (중복 발급)
```

**발생 원인**
```
시간 →
요청1: SELECT 쿠폰(0개) ── 정책검증(✓) ── INSERT ──
요청2: ── SELECT 쿠폰(0개) ── 정책검증(✓) ── INSERT

문제: CouponIssuance Lock은 수량만 보호,
      OnePerMember 정책의 기발급 여부 체크는 Lock 없이 수행됨
```

**비즈니스 영향**
- 회원당 1회 발급 정책 위반
- 마케팅 예산 초과
- 공정성 이슈

## 2. 해결전략

> 🚧 **검토 중**: 각 이슈별 해결 전략은 별도 검토 후 업데이트 예정입니다.

### 2.1 잔액 충전 동시성 제어

**후보 전략**

| 전략 | 설명 | 장점 | 단점 |
|------|------|------|------|
| SELECT FOR UPDATE | 조회 시 row Lock 획득 | 단순, 확실한 보호 | 성능 저하, Lock 경합 |
| 낙관적 락 (@Version) | 충돌 시 예외 발생 | 성능 좋음, Lock 없음 | 충돌 시 재시도 필요 |
| 조건부 UPDATE | 원자적 업데이트 | 단순, 성능 좋음 | 실패 시 재시도 로직 필요 |

### 2.2 재고 감소 동시성 제어

**선택 전략**: 조건부 UPDATE

**선택 근거**:
- 재고 충분 시 불필요한 재시도 방지: 낙관적 락은 version 충돌 시 재고가 충분해도 재시도 필요
- 원자적 처리: `UPDATE ... WHERE stock_quantity >= :quantity` 로 조회-검증-수정을 단일 쿼리로 처리
- 단순한 구조: 추가 컬럼이나 예외 처리 로직 불필요

**구현 방안**:
1. Repository에 조건부 UPDATE 쿼리 추가
2. 영향받은 행이 0이면 재고 부족으로 처리

---

### 2.3 쿠폰 발급 동시성 제어 (OnePerMember)

**후보 전략**

| 전략 | 설명 | 장점 | 단점 |
|------|------|------|------|
| DB Unique 제약 | member_id + coupon_summary_id | 확실한 보호, DB 레벨 | 예외 처리 필요 |
| SELECT FOR UPDATE | 회원 + 쿠폰 조합 Lock | 유연함 | Lock 복잡도 증가 |
| 조건부 INSERT | INSERT ... SELECT WHERE NOT EXISTS | 원자적 처리 | 쿼리 복잡도 |
